SilentSigma 套利者 开发文档


1.项目概述

本项目旨在构建一个基于 **Electron 桌面应用** 的面向加密货币的智能量化交易平台，面向散户投资者，提供从交易所账户绑定（Gate.io）、策略选择、自动交易、收益分析、交易记录的全流程方案。前端采用 Electron 实现跨平台桌面应用，后端采用 FastAPI + Gate.io 官方 Python SDK（gate-api）对接交易所。

项目目标	                         目标用户	         核心功能
为散户投资者提供简单易用的量化交易服务	个人投资者、散户	券商账户绑定、策略选择、自动交易、收益分析、交易记录

·  易用性：界面简洁直观，操作简单易懂
·  安全性：交易数据加密，账户信息安全
·  稳定性：交易系统稳定可靠，7×24 小时服务
·  响应速度：快速的数据更新和交易执行

-关键业务流程说明：

交易所绑定流程：

用户发起绑定 → 选择模式（实盘/模拟）→ 输入 API Key + Secret → 授权交易权限 → 平台验证账户（调用对应 BaseURL）→ 绑定成功 → 同步账户数据 → 开始使用


交易模式切换流程：

用户点击切换 → 检查目标模式是否已绑定 → 已绑定则切换，未绑定则引导至账户设置绑定 → 切换后刷新账户总览、交易记录、持仓 → 实盘切换时弹出风险确认


策略订阅流程：

浏览策略列表 → 筛选策略 → 查看策略详情 → 了解风险收益 → 设置个性化参数（含执行模式：实盘/模拟）→ 确认订阅 → 策略开始运行 → 按所选模式通过对应 API 自动交易执行


交易执行流程：

策略信号产生 → 风控检查 → 按订阅的 mode 选择 broker（实盘/模拟）→ 生成交易指令 → 通过对应 Gate API 下单 → 订单状态监控 → 成交回报处理 → 更新持仓记录 → 生成交易流水


收益分析流程：

数据收集 → 收益计算 → 指标分析 → 可视化展示 → 对比分析 → 风险评估 → 生成报告 → 用户查看

6. Electron 与响应式设计

6.1 Electron 开发要点

· 主进程：窗口创建、菜单、系统托盘、IPC 桥接
· 预加载脚本：通过 contextBridge 安全暴露 API 给渲染进程
· 多窗口：欢迎页、登录页、主平台窗口（单例）
· 打包：electron-builder 生成 Windows/macOS/Linux 安装包

6.2 响应式设计考虑

· 使用 CSS Grid 和 Flexbox 布局
· 媒体查询适配不同屏幕尺寸（窗口可缩放）
· 触摸友好的交互设计（支持触屏设备）

| main       | 是   | 1280×800 最小 | 主平台（仪表盘、账户、策略、个人中心） |

· 主窗口关闭时退出应用；主窗口最小化可进托盘
· 使用 `BrowserWindow` 的 `webPreferences: { nodeIntegration: false, contextIsolation: true }`

6.3.3 主进程入口 (main.js) 职责

· 创建/销毁窗口、应用 ready/quit 处理
· 注册 IPC handlers（委托给 ipc-handlers.js）
· 系统托盘、全局快捷键（可选）
· 不执行重计算，仅做路由与转发


6.4 IPC 设计规范（高性能）

6.4.1 设计原则

· **小数据走 IPC**：< 64KB 的 JSON 可用 `ipcRenderer.invoke`，主进程用 `ipcMain.handle` 返回
· **大数据走 postMessage + Transferable**：K 线、行情等用 `webContents.postMessage` + `ArrayBuffer` 转移所有权，避免 Structured Clone 阻塞
· **渲染进程不阻塞**：耗时解析放入 Web Worker，主线程只做渲染

6.4.2 IPC 通道命名规范

| 通道前缀 | 方向         | 用途                     |
|----------|--------------|--------------------------|
| app:     | 渲染→主      | 应用级（窗口、设置）     |
| store:   | 双向         | 本地存储（偏好、Token）  |
| bridge:  | 主→渲染      | 主进程主动推送（postMessage） |

6.4.3 暴露给渲染进程的 API（preload.js）

```javascript
// 仅暴露必要方法，避免大对象传递
contextBridge.exposeInMainWorld('electronAPI', {
  // 窗口
  app: {
    minimize: () => ipcRenderer.invoke('app:minimize'),
    maximize: () => ipcRenderer.invoke('app:maximize'),
    close: () => ipcRenderer.invoke('app:close'),
    openMain: (token) => ipcRenderer.invoke('app:openMain', token),
  },
  // 本地存储（小数据）
  store: {
    get: (key) => ipcRenderer.invoke('store:get', key),
    set: (key, val) => ipcRenderer.invoke('store:set', key, val),
    // 常用 key：token, trading_mode(real|simulated)
  },
  // 大数据接收：主进程 postMessage 推送，渲染进程监听
  onChartData: (callback) => { /* 见 6.4.5 */ },
});
```

6.4.4 大数据传输（K 线/行情）流程

```
后端 WebSocket → 主进程 Node → ArrayBuffer 解析
    → webContents.postMessage(data, [buffer])
    → 渲染进程 onmessage 接收
    → Web Worker 解析/聚合
    → 主线程 lightweight-charts.update()
```

· K 线增量：主进程只转发 ArrayBuffer，不 JSON 解析
· 历史 K 线：REST 返回后，主进程可做流式转发，或渲染进程直接 fetch

6.4.5 IPC 消息清单

| 通道              | 参数                    | 返回值/行为              |
|-------------------|-------------------------|--------------------------|
| app:minimize      | -                       | -                        |
| app:maximize      | -                       | -                        |
| app:close         | -                       | -                        |
| app:openMain      | token: string           | -                        |
| store:get         | key: string             | any                      |
| store:set         | key, value              | -                        |
| store:trading_mode| -                       | 读取/写入 trading_mode   |
| bridge:chartData  | (主→渲染) symbol, buffer| 无，渲染进程 onmessage   |


6.5 图表与 UI 性能优化规范

6.5.1 K 线图选型：lightweight-charts

· 库：`lightweight-charts`（TradingView 官方，npm install lightweight-charts）
· 特点：约 35KB、Canvas 渲染、支持数千条 K 线实时更新
· 数据格式：`{ time: '2024-01-15' 或 1705305600, open, high, low, close }`

6.5.2 图表性能规范

| 场景           | 策略                                         |
|----------------|----------------------------------------------|
| 初始加载       | 最多加载 500 根 K 线，按需分页拉取更早数据   |
| 实时推送       | 仅 append/update 最新一根，不用 setData 全量替换 |
| 缩放/平移      | 使用图表内置交互，避免自定义拖拽阻塞         |
| 多图同屏       | 单页不超过 3 个图表实例，其余用 Tab 切换     |
| 数据解析       | 在 Web Worker 中解析，主线程只调用 update()   |

6.5.3 UI 响应规范

· 所有网络请求：异步，带 loading 状态，不阻塞 UI
· 列表虚拟滚动：自选币、交易记录等长列表使用虚拟滚动（如 100 条可见）
· 防抖/节流：搜索、筛选输入 300ms 防抖；行情刷新 100ms 节流
· 动画：使用 CSS transform/opacity，避免触发布局重排

13. 量化策略引擎与因子工程

13.1 定位与边界

本平台面向加密货币现货（Gate.io 现货 + 实盘/Testnet 双模式），采用「中频半自动」交易模式：

· 量化引擎只产出**信号 + 建议下单参数 + 风控约束**，由 DeepSeek 审核单做综合二次评估，最终由用户在「订单审核」面板人工点击「通过」后下单。
· 不追求超低延迟撮合，因子计算基于已收盘 K 线（去未来函数），适合 1m / 5m / 15m / 30m / 1h / 4h / 1d 周期。
· 不依赖 Parquet/HDF5 多表历史库；行情与 K 线统一通过 `backend/utils/gate_client.py` 实时拉取并由 Redis（可选）/进程内缓存复用，符合「Gate 加密现货 + 半自动下单」场景。

13.2 引擎模块结构（`backend/services/strategy_engine/`）

| 模块 | 文件 | 职责 |
|------|------|------|
| 因子计算 | `factors.py` | 基础技术因子：动量、反转、波动率、成交量异常、RSI、布林位置、MACD 柱、均线乖离等，输入 OHLCV DataFrame，输出因子矩阵 |
| Alpha158 | `alpha158.py` | 兼容 Qlib 风格的 Alpha158 量价特征生成器，作为 ML 训练特征源 |

| 动态因子执行 | `dynamic_factors_executor.py` | 加载入库的动态因子表达式（`dynamic_factor` 表），编译并对当前数据求值 |
| 权重合成 | `weights.py` | 按各因子近期 ICIR 做 softmax 归一化权重，对标准化因子加权合成综合得分 |

13.3 因子库分层

引擎内部因子按来源分为三层，按优先级合并到综合得分：

1. **静态基础因子**：`factors.py` 中硬编码的常用技术因子；总是参与计算，作为兜底信号。
2. **Alpha158 量价因子**：`alpha158.py` 输出，主要用于 ML 模型训练（不直接参与线性合成）。
3. **动态因子（DeepSeek 挖掘入库）**：通过「因子挖掘」入口由 DeepSeek 生成 → 用户审核 → 写入 `dynamic_factor` 表 → `dynamic_factors_executor.py` 在每次推理时按入库表达式求值；同时 `factor_library_refresh_worker.py` 周期性刷新其 IC/ICIR，并按阈值停用失效因子。

13.4 信号生成流程

```
Gate K 线 (gate_client)
   ↓
factors.py + dynamic_factors_executor.py 计算因子矩阵
   ↓
factor_evaluation.py 计算近窗口 IC/ICIR
   ↓
weights.py softmax(ICIR) → 归一化权重
   ↓
综合得分 = Σ w_i × zscore(factor_i)            （线性合成）
ml_model.py 输出 p_up                           （可选 ML 增强）
   ↓
position_risk.py：ATR 止损止盈 / Kelly 上限 / 仓位比例约束
   ↓
strategy_engine 包：{score, p_up, suggested_order, stop_loss, take_profit, amount, …}
   ↓
order_audit_service：合并账户、持仓、风控偏好 → 调用 DeepSeek 生成审核 JSON
   ↓
前端「订单审核」列表 → 用户「通过」 → 按审核单中的方向/价格/数量执行 Gate 现货下单
```

13.5 去未来函数与去过度拟合约束

· **严格只用历史**：因子在第 t 根上的取值仅依赖 ≤ t 的数据；ML 训练时标签使用 `close[t+1]` 但特征使用 `≤ t` 信息，预测阶段只输入 ≤ 当前根特征。
· **滚动窗口验证**：`factor_evaluation.py` 用滚动 N 期窗口估计 IC/ICIR，避免一次性全样本调参。
· **正则与早停**：ML 模型设置 `max_depth`、L1/L2 正则；`scikit-learn` 默认开启交叉验证选参。
· **稳健性检验**：回测同时报告夏普、最大回撤、胜率、盈亏比、换手率，要求夏普 > 1、最大回撤 < 20%、盈亏比 > 1.5（仅作为筛选阈值，前端展示真实数值）。

13.7 风险评估与仓位计算

`position_risk.py` 实现以下规则（数值可在「风险设置」页配置）：

· **ATR 止损止盈**：止损价 = 开仓价 − ATR × `stop_atr_mult`；止盈价 = 开仓价 + ATR × `take_atr_mult`；做空对称。
· **简化 Kelly**：`f* = (p·b − q) / b`，p=胜率，b=盈亏比；上限裁剪为 `max_kelly`（默认 20%）。
· **波动率定标仓位**：仓位 = 目标波动率 / 预期波动率（来自近 N 根收益率标准差），上限受账户最大持仓比例约束。
· **账户级硬约束**：单标的不超过 `max_position_pct`，组合杠杆 ≤ 1（现货），未撮合订单数量上限。
· 实盘与模拟可在 `risk/settings?mode=...` 中独立配置一套风控参数。

14. 因子评估体系（`factor_evaluation.py`）

评估一个因子是否「合格」需从下面五个维度系统检验，单维度通过不代表整体可用：

14.1 预测能力

· **信息系数（IC / Rank IC）**：当期因子值与下一期收益的（秩）相关系数。
  - 阈值：|IC Mean| > 0.02 且 ICIR = mean(IC) / std(IC) > 0.5 视为有效；
  - 稳健策略偏好高 ICIR、低 IC 波动；积极策略可容忍 IC 波动大但要求极端组超额收益显著。
· **分层回测**：按因子值排序分 5 组，要求最高组与最低组收益显著分化、且中间组单调；非单调的因子需谨慎。

14.2 稳定性与衰减

· **时间序列稳定性**：跨牛/熊/震荡市的 IC 一致性；以「半衰期」（IC 衰减到一半所需时间）衡量信号持续性。
· **换手率 / 自相关性**：因子自相关 > 0.9 → 慢变量，适合低频、覆盖交易成本能力强；< 0.5 → 快变量，必须严格控制成本。

14.3 独立性与增量贡献

· **相关性矩阵**：新因子与既有因子池的平均相关 > 0.7 视为冗余。
· **增量 IC**：在控制现有因子（逐步回归 / 条件 IC）之后，新因子是否仍显著。
· 实现：`factor_evaluation.py` 输出每个候选因子相对于已入库因子的剩余 IC 与 VIF。

14.4 经济逻辑与可解释性

· **逻辑自洽**：拒绝纯数据挖掘的高 IC 因子；要求金融/市场机制解释（如「资金费率过高 → 多头拥挤 → 后续下跌概率」）。
· **机器学习可解释**：在 ML 综合阶段使用 SHAP 值审视因子贡献；黑盒因子或仅靠少量样本支撑的 SHAP 高分一律降权。

14.5 考虑成本后的可行性

· **盈亏比 + 胜率 + 成本**：单次成交成本（手续费 + 滑点）按 0.05% ~ 0.1% 计；50% 胜率 × 1:1 盈亏比扣除成本必亏，需用真实成本做净化筛选。
· **容量与流动性**：加密订单簿深度有限，大资金需评估因子在该标的上的容量衰减（订单簿深度采样）。

14.6 入库筛选流程

`factor_library_refresh_worker.py` 后台定时执行以下管线：

1. **单因子初筛**：IC、ICIR、分层单调性、半衰期；
2. **稳健性检验**：参数（计算窗口、持有期）扰动 + 时段切片；
3. **多因子共线性处理**：保留低相关、高 ICIR 的子集；
4. **组合测试**：纳入综合因子，对比夏普 / 最大回撤 / 盈亏比是否提升；
5. **入库 / 失效**：通过则写入 `dynamic_factor` 表（`status=active`）；既有因子若连续 N 期 ICIR 不达标则降级为 `inactive`，并写入 `factor_library_refresh_job` 任务记录。

15.4 安全与可控性约束

· **沙箱化执行**：DeepSeek 生成的因子表达式必须通过 `dynamic_factors_executor.py` 的 AST 白名单（仅允许指定字段、算术运算、`shift / rolling / pct_change / ewm` 等）；禁止 `eval/exec/import` 等。
· **未来函数检查**：编译期检查所有 `shift(n)` 必须 `n >= 0`（不允许 `shift(-1)` 等向未来读取的负偏移）。
· **缓存表达式 → 代码**：相同表达式不重复调 API；表达式与生成代码做哈希对应，存数据库。
· **数据脱敏**：仅向 API 传字段名、统计量、IC 序列等聚合信息，绝不上传原始账户/持仓/订单数据。
· **API Key 安全**：每个用户的 DeepSeek Key 加密存储于 `user_preferences`（key 字段），仅 `deepseek_service.py` 内部使用。

15.5 自动化 Agent 闭环（已落地的最小版本）

`factor_library_refresh_worker.py` 后台周期任务实现了「生成器 → 执行器 → 评估器 → 决策器」最小闭环：

1. 定时调用 `deepseek_factor_agent` 生成 K 个候选；
2. 对每个新因子在最近窗口数据上跑评估；
3. `deepseek_factor_mining_agent` 综合评估结果 + 表达式逻辑给出 `keep / improve / drop`；
4. `keep` → `dynamic_factor` 表 active；`improve` → 写入 `factor_library_refresh_job` 待下一轮再生成；`drop` → 记录失效原因；
5. 既有 active 因子定期再评估，连续不达标则 → inactive。

| `POST /api/v1/strategy-engine/factor/mine` | 立即触发一次挖掘流水（产生候选 + 评估 + 入库决策） |

16. 真实 Brinson 归因与正式风险模型（`backend/services/analytics/`）

16.1 目标

替换早期「因子权重 × ICIR 代理」式的伪归因和「单标的波动」式的伪风险评估，引入：

· **Brinson-Fachler 多期归因**：使用对齐的「组合权重 / 基准权重 / 资产收益」三矩阵做配置 / 选股 / 交互效应分解；
· **结构化风险模型**：协方差分解 Σ = B F Bᵀ + D（B = 暴露矩阵、F = 因子协方差、D = 特异风险），输出组合波动、跟踪误差、边际风险贡献、VaR 等。

16.2 模块结构

```
backend/services/analytics/
├── brinson.py        # Brinson-Fachler 归因（含行业聚合）
├── risk_model.py     # 简化 Barra 风格风险模型
├── panel.py          # 时间网格 / 三矩阵对齐工具
├── utils.py          # 重采样、winsorize、协方差估计辅助
├── config.py         # 基准定义、行业映射、估计参数
└── __init__.py
```

16.3 数据准备：三矩阵对齐

`panel.py` 负责把回测和实盘运行结果整理为统一时间网格上的三个 DataFrame：

| 矩阵 | 形状 | 含义 | 来源 |
|------|------|------|------|
| `weights_portfolio` | (T, N) | 组合权重，行和为 1 | `portfolio_report.run_portfolio_visual_backtest` 的 `weights_history` |
| `weights_benchmark` | (T, N) | 基准权重 | `analytics/config.py` 中定义的基准（等权 / 流动性前 N / 固定比例 BTC40+ETH30+其他30） |
| `asset_returns` | (T, N) | 资产收益 | Gate K 线收盘价计算的区间收益 |

行业映射 `sector_map`（`config.py`）：手动维护 `symbol → sector` 表（Layer1 / DeFi / Stablecoin / Meme / 主流币 / Other …），可后续扩展为 CoinGecko 标签对接。

16.4 Brinson-Fachler 归因（`brinson.py`）

实现 `brinson_fachler(weights_portfolio, weights_benchmark, asset_returns, sector_map=None)`，输出：

| 字段 | 类型 | 说明 |
|------|------|------|
| `allocation_effect` | Series (T,) | 配置效应 = Σ_sec (w_p − w_b) × r_b（用基准收益） |
| `selection_effect` | Series (T,) | 选股效应 = Σ_sec w_b × (r_p − r_b)（行业内组合 vs 基准收益） |
| `interaction_effect` | Series (T,) | 交互效应 = Σ_sec (w_p − w_b) × (r_p − r_b) |
| `total_effect` | Series (T,) | 组合收益 − 基准收益（应等于上述三项之和） |
| `summary` | dict | 累计效应、信息比率、按行业分解的贡献排名 |

· 单层 Brinson：`sector_map=None` 时直接按 symbol 当作各自的 sector。
· 行业层 Brinson：`sector_map` 非空时按行业聚合权重与收益再分解。
· 多期链式合成：使用 GRAP 几何链式法以避免简单线性求和带来的偏差。

16.5 风险模型（`risk_model.py`）

最小可行版本（适合加密现货数据量级）：

· **因子定义**：3~5 个常见因子（市场收益、动量、波动率、成交量异常、行业哑变量）；可在 `analytics/config.py` 中扩展。
· **B 估计**：横截面回归 r_i = α_i + B_i f + ε_i，窗口 60~120 个 bar；指数加权（EWMA）。
· **F 估计**：因子收益历史协方差，半衰期可配置。
· **D 估计**：横截面回归残差方差。

输出：

```python
risk_metrics = {
    "portfolio_vol":        # 事前组合波动率（年化）
    "tracking_error":       # 相对基准跟踪误差
    "factor_contributions": # 各因子贡献占比
    "asset_mrc":            # 边际风险贡献 Top N
    "var_95": / "cvar_95":  # 95% VaR / CVaR
    "max_drawdown":         # 真实回测最大回撤（保留）
}
```

16.6 与回测/报告的协同

`portfolio_report.py` 中 `run_portfolio_visual_backtest` 与 `build_analytics_report` 协同：

```python
nav, weights_df, returns_df = run_portfolio_visual_backtest(...)
# weights_df / returns_df 已是 T×N 对齐矩阵

if config.use_brinson:
    brinson_res = brinson_fachler(weights_df, get_benchmark_weights(...), returns_df, sector_map)
else:
    brinson_res = None  # 降级为只展示因子 ICIR

```

回测页前端读取该报告，将归因效应、行业贡献、风险摘要分别渲染为表格 + 图表。

18. 一键选币（仪表盘 · 选币栏）

18.1 功能概述

「一键选币」位于**仪表盘**的**选币栏**顶部，从 Gate.io 全市场 USDT 现货交易对中自动筛选少量优质标的，帮助用户快速构建自选池。

| 维度 | 说明 |
|------|------|
| 输入 | Gate.io 全市场现货 ticker（数百至上千个 USDT 交易对） |
| 输出 | 默认 10 个（可调，3~15）优质币种 |
| 筛选维度 | 流动性（24h 成交额）、价格下限、24h 涨跌幅区间、（DeepSeek 介入时）综合语义评估 |
| 触发位置 | 仪表盘选币栏右上角「选币」「接入Agent」按钮 |

18.2 实际架构（已收敛为两层）

```
              用户点击「选币」 / 「接入Agent」
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ 第一层：规则引擎（rule_engine.py）                        │
│  - 仅 USDT 现货交易对                                    │
│  - 24h 成交额 ≥ min_quote_volume（默认 50,000 USDT）     │
│  - 价格 ≥ min_price（默认 1e-8）                         │
│  - |24h 涨跌幅| ≤ max_change_24h（默认 1.0 = 100%）      │
│  - 输出按成交额降序的候选池（含 reason）                  │
└─────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┴───────────────────┐
        ▼                                     ▼
┌────────────────────────┐       ┌──────────────────────────────────┐
│ 「选币」直接返回 Top N │       │ 「接入Agent」（DeepSeek）         │
│ source=rule_engine     │       │ 取规则池前 55 个 → DeepSeek 选 N │
│                        │       │ source=deepseek_agent            │
└────────────────────────┘       └──────────────────────────────────┘
                          │
                          ▼
              前端弹窗展示候选 + 理由
                          │
                          ▼
              用户勾选 → /dashboard/watchlist/batch
                          │
                          ▼
                  自选列表实时刷新
```

> **与早期设计的差异**：早期文档规划的「ML 模型层（XGBoost/LightGBM `coin_scorer.pkl`）」已不再使用，原因是：① 加密市场分布漂移快、监督标签难标注；② 规则引擎成交额排序已能给出稳定可靠的流动性 Top N；③ 语义层与偏好理解能力交由 DeepSeek 完成，避免冗余。当前架构以「确定性规则 + LLM 语义增强」替代「规则 + ML + LLM」的三层设计。

18.3 前端表现（仪表盘选币栏）

┌─────────────────────────────────────────────────────────┐
│  选币                                                    │
│  ┌──────────┐  ┌──────────────┐  ┌────────┐            │
│  │ 选币     │  │ 接入Agent    │  │ 刷新   │            │

| 按钮 | 行为 | 备注 |
|------|------|------|
| **选币** | 调用 `/dashboard/smart-select` → 弹窗展示候选 → 全选/勾选 → 批量加入自选 | 无需任何 API Key，纯规则引擎 |

18.4 规则引擎（`backend/services/rule_engine.py`）

```python
DEFAULT_RULES = {
    "min_quote_volume": 50_000,   # 24h 成交额最低 5 万 USDT
    "max_change_24h":   1.0,      # |24h 涨跌幅| ≤ 100%
    "min_price":        1e-8,     # 最低价格
}

def apply_rules(tickers, mode="real", rules_override=None) -> list[dict]:
    """对 Gate ticker 列表应用规则，返回按成交额降序的候选 [{symbol, volume, change_pct, last, reason}, ...]"""
```

实现要点：

· 仅保留 `currency_pair` 含 `_USDT` 的交易对；
· 自动适配 Gate API 返回的 `change_percentage` 单位（>1 时按百分比折算回小数）；
· 当三条规则都过严导致候选为空时，自动放宽为「所有 USDT 对按成交额排序」做兜底，保证「选币」按钮永不返回空列表；
· `reason` 字段根据流动性 + 涨跌方向自动生成（如「高流动性、趋势向上」「高流动性、短期回调」）；
· 规则数值可由前端通过请求体覆盖（详见 18.5 接口）。

18.5 后端 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/dashboard/smart-select-rules` | GET | 返回当前默认规则（`{min_quote_volume, max_change_24h, min_price}`），用于规则调节面板 |
| `/api/v1/dashboard/smart-select`       | POST | 一键选币：纯规则引擎 |
| `/api/v1/dashboard/agent-select`       | POST | DeepSeek Agent 选币（需 DeepSeek Key） |

**18.5.1 `POST /dashboard/smart-select`**

请求体（全部可选）：

```json
{
  "top_n": 10,
  "mode": "real",
  "min_quote_volume": 50000,
  "max_change_24h": 1.0,
  "min_price": 0.00000001
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| top_n | int | 返回数量，默认 10 |

```json
{
  "success": true,
  "data": {
    "symbols": [

    ],
    "source": "rule_engine"
  },
  "message": "ok",
  "code": 200
}
```

**18.5.2 `POST /dashboard/agent-select`**

请求体：

```json
{ "preference": "偏稳健，偏好主流币", "top_n": 10, "mode": "real" }
```

| 字段 | 类型 | 说明 |
|------|------|------|
| preference | string | 自然语言偏好；为空时由 DeepSeek 自由发挥 |
| top_n | int | 3 ~ 15，默认 10（后端硬裁剪到该区间） |
| mode | string | 同上 |

后端流程（详见 `backend/api/dashboard.py::agent_select`）：

1. 从 `user_preferences` 读取该用户的 DeepSeek API Key；未绑定时返回 `{success:false, data:{needs_deepseek:true}, code:400}`，前端引导跳转个人中心绑定。
2. 拉 Gate ticker → `apply_rules` → 取规则池前 55 个作为候选。
3. `services/deepseek_coin_agent.run_agent_coin_pick` 调用 DeepSeek（强制 JSON Object 模式），强约束 `symbol` 必须来自候选列表。
4. 模型未凑满 `top_n` 时按候选顺序补齐，保证返回数量稳定。

响应：

```json
{
  "success": true,
  "data": {
    "symbols": [{ "symbol": "BTC_USDT", "reason": "流动性最佳，趋势稳健" }],
    "summary": "基于当前市场，推荐以上币种...",
    "source": "deepseek_agent"
  },
  "message": "ok",
  "code": 200
}
```
